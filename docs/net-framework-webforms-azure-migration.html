Migrating a 20-Year-Old .NET Framework WebForms Platform to Azure

A practical, decision-oriented guide for rehosting a large, long-lived ASP.NET WebForms application — mixed VB.NET and C#, LINQ to SQL, a Windows Service scheduler, in-memory workflows, and hundreds of cross-database dynamic SQL statements — onto Azure. Evaluates the VM path (least effort) against Azure App Service with and without serverless (FaaS) components, and proposes a phased plan.

Read this first — the one decision that shapes everything else

For this class of application the critical migration decision is not the web tier. It is the data tier. The application runs hundreds of dynamic SQL statements and stored procedures whose object names contain three- and four-part names ([OtherDb].[dbo].[Table], sometimes [Server].[Db].[dbo].[Table]) across many databases. Azure SQL Database (the single-database PaaS) does not support cross-database queries by three/four-part name — rewriting hundreds of statements to elastic-query external tables is a project in itself and a regression minefield. Azure SQL Managed Instance supports cross-database queries, linked servers (to SQL-family targets), SQL Agent and Database Mail — meaning the entire T-SQL estate can move unchanged if all databases land on the same instance. Choose the data tier first; the web-tier choice (VM vs App Service) then becomes a much smaller, reversible decision.

1. The system profile

The subject is a web platform, roughly two decades old, operated by an international scientific organisation for the submission, aggregation and statistical estimation of field-sampling data. Its architecture is typical of well-built enterprise .NET applications of its era — which means it embodies, precisely, the set of assumptions the cloud breaks. Every migration decision below traces back to one of these traits:

Out of scope for this guide: rewriting the UI. ASP.NET WebForms has no path to modern .NET (no .NET 8/9/10 support), and VB.NET has no WebForms successor. A future modernisation (e.g., Blazor) is a separate programme; everything below keeps the app on .NET Framework 4.8 — which remains supported as a Windows component — and moves the infrastructure to Azure.

Anti-pattern to avoid: migrating the app and "modernising" it in the same step

With hundreds of dynamic SQL statements and a 20-year change history, the migration must be behaviour-preserving and diffable: same inputs → same outputs, before and after. Every deliberate change you mix into the move (new mapper, new session store, rewritten workflow) makes a regression impossible to attribute. Move first with minimal touches; modernise afterwards on a stable cloud baseline.

3. Trait-by-trait cloud impact analysis

This is the heart of the assessment: what each trait means on a VM, on App Service, and what (if anything) must change. Severity is from the App Service perspective — on a VM, nearly everything "just works".

T1 — Mixed VB.NET + C# on .NET Framework 4.8 LOW RISK

Both languages compile to the same IL; App Service (Windows) and any Windows VM run .NET Framework 4.8 natively. There is nothing to do for the migration itself. Two forward-looking notes:

  • Ensure everything targets 4.7.2+ (ideally 4.8): TLS 1.2+ then follows OS defaults — required by Azure SQL and most Azure endpoints. A 4.5-era project pinned to old TLS behaviour is the classic "cannot connect after migration" cause.
  • Keep the solution buildable from a clean machine (NuGet restore, no GAC-only references) — you will need clean CI builds for slot-based deployment later. Inventory any GAC/COM dependency now (see appendix); those are the only T1 items that can genuinely block App Service.

T2 — LINQ to SQL LOW RISK

LINQ to SQL is legacy but fully functional on .NET Framework 4.8 and works unchanged against Azure SQL Managed Instance or SQL Server on a VM. Do not convert it to EF as part of the migration (see anti-pattern above). Practical points:

  • Connection strings move to App Service configuration (or Key Vault references) — App Service overrides web.config connection strings by name at runtime, so no config-file surgery is needed.
  • Add Encrypt=True; Azure SQL endpoints use CA-signed certificates, so TrustServerCertificate should not be needed (unlike self-signed on-prem instances).
  • Start with SQL authentication (secret in Key Vault); adopt Microsoft Entra authentication as a hardening step later — System.Data.SqlClient on .NET Framework supports it, but don't couple it to the move.
  • Watch for DataContext instances cached beyond a request (they sometimes hide inside T3's cache) — a cached DataContext holds a connection's identity map and behaves badly under recycling. Inventory, don't refactor yet.

T3 — HttpContext.Current / in-memory caching MEDIUM RISK

Everything cached in process memory carries three cloud assumptions: (1) there is one web server, (2) it doesn't restart often, (3) background code runs inside a request so HttpContext.Current is non-null. On App Service:

  • Scale-out breaks assumption 1. Two instances = two independent caches = users see stale/different data depending on which instance served them. Mitigation for the move: run a single instance and switch ARR affinity (session cookie stickiness) on — this is a perfectly legitimate transitional stance for an internal/expert-user system. Distributed cache (Azure Cache for Redis) is the later modernisation, not a migration prerequisite.
  • Recycles break assumption 2. App Service worker processes restart on platform maintenance, config changes and slot swaps — more often than a hand-tuned on-prem IIS. The app already survives nightly IIS recycles on-prem, so a cold cache must already be tolerable; verify that cache population is lazy and idempotent, and enable Always On so the site isn't unloaded when idle.
  • Background extraction breaks assumption 3. Any code you later move out of the request pipeline (T6) will find HttpContext.Current == null. When extracting workflows into WebJobs, pass state explicitly; grep for HttpContext.Current in the business layer now to size that work (script in the appendix).
  • Session state: WebForms wizards almost certainly use InProc session. Same story: fine on one sticky instance; for scale-out or resilience move to the SQL Server session-state provider (a natural fit — the session DB simply becomes one more database on the Managed Instance) or the Redis session provider. Both require session-stored objects to be serializable — inventory that before switching.
  • Pin the <machineKey>. On a single on-prem server the auto-generated machine key never mattered. In App Service, restarts/swaps regenerate auto keys → ViewState MAC failures and forms-auth logouts. Generate one explicit <machineKey> and put it in web.config before the first deployment. This is the single most common WebForms-on-App-Service gotcha.

T4 — Windows Service running weekly SQL jobs LOW RISK — with the right target

App Service cannot host a Windows Service. But look at what the service actually does: it triggers SQL work on a schedule. That is exactly what SQL Server Agent does — and SQL Agent exists on Azure SQL Managed Instance (T-SQL job steps supported, schedules supported, Database Mail notifications supported with the AzureManagedInstance_dbmail_profile profile). Options, best first:

Triggered WebJob (CRON) Convert the service body to a .NET Framework console app, deploy it as a scheduled WebJob beside the web app (requires Always On). If the service does more than SQL (file shuffling, calls into app DLLs). WebJobs run full .NET Framework — legacy code moves as-is.

T5 — Import page → API handle → poll a DB table LOW RISK — it's already cloud-shaped

This is the pleasant surprise of the profile: twenty years ago somebody built a textbook asynchronous request–reply pattern — kick off work, get a handle, poll for completion, resume. That is exactly the pattern Azure prescribes for long-running operations. Consequences:

  • The pattern survives migration unchanged: the API moves with the app (or stays wherever it lives), the status table moves with its database, the polling loop keeps working.
  • Verify how the page polls. If it polls client-side (meta-refresh / AJAX timer hitting the server every few seconds), each poll is a short request — perfect. If it blocks server-side inside one request while polling, that request is subject to the 230-second limit (T6) and must be converted to client-side polling — a small, page-local change.
  • The "resume and continue processing the file" step after completion is a candidate for the same treatment as T6 if it can exceed a few minutes: enqueue it, reuse the very same status-table + polling UX the users already know. Reusing the app's own established pattern is the lowest-risk fix for every long operation in the system.
  • Uploaded files: store them under %HOME% (persistent, shared, Azure-Files-backed) or better, in Blob Storage with the DB row holding the blob path. Avoid %TMP% for anything that must survive a recycle — it's per-instance and volatile.

T6 — 10-minute in-memory WebForms workflows HIGH RISK on App Service — the one true blocker to engineer around

The 230-second wall

App Service (Windows) terminates any HTTP request that hasn't returned a response after ~230 seconds — a platform front-end/load-balancer limit that cannot be raised (Linux: 240s; raising ASP.NET's own executionTimeout in web.config does not help — the platform cuts the connection first). A synchronous postback that computes for 10 minutes will fail with a 502 on App Service, no matter the plan tier. On a VM you control IIS end-to-end and today's behaviour continues — this single trait is why the VM path is "least effort".

Engineering around it, in increasing order of invasiveness:

  1. Inventory which operations actually exceed ~3 minutes. Often only a handful of steps do. Everything under the limit needs no change. (IIS logs / App Insights on the current system give you the histogram.)
  2. If the long step is really the database working (a heavy proc call awaited synchronously): switch that page to the T5 pattern the app already owns — insert a work request row, have the work run in the background (SQL Agent job on demand, or a WebJob picking up the queue table), return immediately, poll client-side, resume on completion. UI-wise users see the same "import-style" progress experience they already know.
  3. If the long step is app-tier computation on in-memory data: extract it into a continuous WebJob consuming a queue (a DB table queue keeps it simple and transactional; Azure Storage/Service Bus queues are the later refinement). The WebJob is a .NET Framework console app referencing the same VB.NET/C# business DLLs — the computation code moves nearly verbatim; what changes is marshalling: input state must be persisted (DB/Blob) instead of assumed in session, and HttpContext.Current references inside the moved code must be replaced with explicit parameters (T3).
  4. Transitional stopgap (use sparingly): break the computation into chunks of < 3 minutes driven by successive client-side calls. It avoids background infrastructure but leaves fragile long-running state in session; acceptable to de-risk one stubborn page, not as the general answer.

Also remember: even on a VM, a 10-minute browser request is fragile (proxies, laptops sleeping, impatient F5). The queue-and-poll conversion is worth doing eventually regardless of hosting.

T7 — Hundreds of dynamic SQL statements with multi-part names DECISIVE for the data tier

The "many dots" are three/four-part names: [Database].[schema].[object] and occasionally [LinkedServer].[Database].[schema].[object]. Because they are embedded in dynamic SQL and proc bodies, they are effectively unrefactorable at migration scale — any target that doesn't execute them verbatim forces a rewrite of unknown breadth. This eliminates Azure SQL Database (no cross-DB three-part names; elastic query is read-only external tables and a per-statement rewrite) and makes the decision for you:

  • All databases must land on one SQL Managed Instance (or one SQL Server VM). Then every [OtherDb].[dbo].[Table] reference keeps working, byte for byte.
  • Four-part (linked-server) names need inventory: MI supports linked servers to SQL-family targets (SQL Server, MI, Azure SQL DB, Synapse SQL) — but not to files, Access, Oracle or other OLE DB sources. If a linked server points outside the SQL family, that integration needs a replacement (import job, data virtualization via OPENROWSET on Blob, or a small ETL) — find these early (appendix query).
  • Collation is a silent killer for dynamic SQL. The MI server collation is chosen at creation and cannot be changed. If it doesn't match the on-prem server collation, every temp-table join in those hundreds of statements risks Cannot resolve the collation conflict errors (tempdb inherits server collation). Read the current server collation first (appendix) and create the MI with exactly that.
  • Same story for the time zone: MI lets you pick the instance time zone only at creation. A 20-year-old app is full of GETDATE()/DateTime.Now assumptions; matching the on-prem time zone avoids a whole class of "the weekly job ran at the wrong hour / dates shifted by two hours" regressions. (On App Service set WEBSITE_TIME_ZONE to match; on a VM set the OS time zone.)

T8 — Many databases LOW RISK once T7 is settled

  • Consolidate all of them onto the single Managed Instance in one migration wave. A split data estate (some DBs moved, some not) turns every cross-DB query into a cross-network query — latency and correctness chaos. Move the data tier atomically.
  • Cross-database transactions work within the instance; if any code uses distributed transactions (MSDTC) across servers, MI supports DTC scenarios but flag them for explicit testing.
  • Watch security model details: cross-DB ownership chaining and TRUSTWORTHY settings, server-level logins mapped across DBs, SQL users with server roles — script logins/permissions across (e.g., sp_help_revlogin-style transfer) and verify EXECUTE AS paths used by procs.
  • Size the instance for the sum of all databases (storage and IO). General Purpose MI has per-file IO characteristics; the weekly batch (T4) is often the IO peak that defines the tier, not the daily web load.

Recommendation

Azure SQL Managed Instance, all databases on one instance, provisioned with the on-prem server's exact collation and time zone. It is the only PaaS target that runs the T-SQL estate unchanged, absorbs the Windows Service via SQL Agent, and removes database ops from the team. SQL-on-VM is the fallback if assessment uncovers hard blockers (non-SQL linked servers you can't replace, CmdExec job steps, filesystem-dependent CLR, xp_cmdshell usage). This data-tier choice is identical for Option A and Option B — which is what makes the web-tier decision low-stakes and even reversible.

Assessment tooling

  • Run the Azure SQL migration assessment (Azure Data Studio migration extension / Azure Migrate database assessment) against the source instance — it flags MI-incompatible features per database automatically.
  • Complement it with the appendix queries: they specifically hunt what generic tools under-report — cross-DB references inside dynamic SQL, linked-server targets, Agent job step types, collation, and deprecated-feature counters.
  • Rehearse with native backup/restore to a test MI early. Restoring all databases and running the test ladder against them answers 80% of the risk questions for the price of one weekend.

What it involves

  1. What you gain over the VM

  • No OS to own. Patching, TLS config, IIS care — gone.
  • Deployment slots: stage a build, warm it, swap; swap back = instant rollback. (Mind: swaps recycle the worker — schedule around long-running workflow sessions, another reason to finish constraint #1 properly.)
  • Application Insights auto-instrumentation for .NET Framework — request/dependency/SQL telemetry with zero code changes, toggled in the portal. For a 20-year-old app this observability jump is transformative: it's how you'll find every slow dynamic query the day after cutover.
  • A path to scale-out once state is externalised (constraint #2's second step), plus platform HA for the web tier immediately.

Sizing & settings starting point

Plan: P1v3 (one production instance) — scale the SKU up rather than out until state is externalised. Settings: Always On = On, ARR affinity = On, 64-bit worker, WEBSITE_TIME_ZONE = on-prem time zone, App Insights on, VNet integration to reach the MI privately, connection strings + secrets as Key Vault references. Deploy via ZIP/Run-from-package from CI, not FTP.

7. Option B+ — where FaaS genuinely helps (and where it doesn't)

The .NET Framework rule for Functions

Modern Azure Functions runtimes do not host .NET Framework class libraries — you cannot lift the VB.NET/C# business DLLs into a Function. Functions in this architecture must be thin, new, modern-.NET components that call stored procedures or HTTP endpoints. Anything that needs the legacy DLLs belongs in WebJobs (full .NET Framework console apps) — that's the pragmatic FaaS boundary for this system. Also note the platform symmetry: an HTTP-triggered Function has the same 230-second response limit; FaaS is not an escape hatch from constraint #1 — Durable Functions implement the same async-handle-poll pattern the app already uses.

FaaS componentReplaces / addsWorth it?
Timer-triggered Function calling the weekly procsAlternative to SQL Agent / scheduled WebJob (T4) Only if you standardise all scheduling on Functions; otherwise SQL Agent is closer to the data and simpler. OPTIONAL
Durable Functions orchestration for the import chain (T5): upload → validate → process → notifyThe DB-polling choreography, eventually Attractive after the move, when the import API contract is being touched anyway. Durable's status API is a managed version of the handle-and-poll table. PHASE 4

Bottom line: ship Option B with SQL Agent + WebJobs (zero new runtimes), and adopt FaaS selectively in the hardening phase where each piece pays rent. FaaS is a modernisation dividend, not a migration requirement.

Recommended course

Data tier (both options): SQL Managed Instance, all databases, matching collation & time zone. Weekly jobs to SQL Agent, retiring the Windows Service.

Web tier: aim for Option B (App Service, single sticky instance, WebJobs), gated on one honest spike: the T6 inventory. If only a handful of pages exceed ~3 minutes, convert them using the app's own polling pattern and go App Service — the ops savings compound forever. If the inventory reveals long-request behaviour woven through dozens of workflows, take Option A now (same MI underneath) and schedule the App Service replatform as a second step — nothing about the VM path is wasted, since the data-tier work, config externalisation and test ladder all carry over.

9. The phased migration plan

Phase 0 — Inventory & assessment (2–4 weeks, no infrastructure)

  • Run every script in the appendix: cross-DB references, linked servers, Agent job steps, collation/time zone, dynamic-SQL census, HttpContext.Current/session/static-state census, SMTP, file-path and COM/GAC census.
  • Run the Azure SQL migration assessment against all databases; triage findings into blocker / needs-change / noise.
  • Build the T6 histogram: from IIS logs, list pages by max request duration. Decide Option A vs B on evidence.
  • Freeze the target: region (one region for app + MI — the app is chatty; latency between tiers must be sub-millisecond-ish, which also rules out any "app on-prem, DB in Azure" interim), MI SKU (sum of DB sizes + IO headroom for the weekly batch), App Service plan.
  • Stand up CI that builds the solution cleanly from source. You cannot deploy what you cannot build.

Phase 1 — Data-tier rehearsal (the cheap de-risker)

  • Provision a test MI with the matched collation/time zone. Restore all databases via native backup from URL.
  • Re-create Agent jobs; run the weekly job by hand; diff outcomes against on-prem.
  • Spin up a throwaway test VM in the same VNet, deploy the app on it, point it at the test MI, and run the test ladder. This exercises the hundreds of dynamic statements against MI with zero commitment to the final web tier — the single highest-value rehearsal available, and it works identically whether you end up choosing Option A or B.
  • Fix what falls out (collation conflicts, login mappings, ownership chains, non-SQL linked servers).

Phase 2 — Web tier build-out (Option B shown; Option A is the same minus conversions)

  • App Service plan + web app; settings per §6; pin <machineKey>; VNet integration; Key Vault references; App Insights on.
  • Convert the T6 pages identified in Phase 0 (async-handle-poll + WebJob worker); convert server-side-blocking polls in T5 to client-side polling if found.
  • Windows Service → SQL Agent job(s); delete the service from the deployment; if any non-SQL logic remains, a triggered WebJob console app inherits it.
  • File I/O → %HOME% or Blob; SMTP → authenticated 587/ACS; remove any writes outside the app root.
  • Deploy to a staging slot; run the full test ladder there against the test MI.

Phase 3 — Cutover

  • Announce a freeze window (submissions paused — the async import queue makes this graceful).
  • Final data sync to the production MI: Managed Instance link or fresh native restores during the freeze.
  • Swap staging → production slot (or flip DNS to the VM); smoke-test the ladder's top rungs.
  • Keep the on-prem system read-only warm for a defined rollback window (e.g., two weeks): rollback = DNS back + restore the delta or replay the frozen queue.

Phase 4 — Hardening & dividends (post-stability)

  • Externalise session (SQL provider on the MI — it's just one more database) and cache (Redis) → enable scale-out and remove ARR affinity.
  • Adopt FaaS where it pays (§7); consider Service Bus for the import pipeline.
  • Microsoft Entra authentication (App Service Easy Auth in front of forms auth, or proper integration), private endpoints everywhere, CI/CD with slot gates.
  • Only now, with telemetry in hand, consider the long-term UI modernisation programme — as a separate, data-informed decision.

13. Appendix — inventory & assessment scripts

A. SQL estate census (run on the source server)

-- A1. Server collation & time zone context (MI: both are CREATION-TIME choices)
SELECT SERVERPROPERTY('Collation') AS server_collation;
SELECT name, collation_name FROM sys.databases ORDER BY name;

-- A2. Cross-database references visible to the engine (static SQL in modules)
SELECT DB_NAME() AS from_db, OBJECT_SCHEMA_NAME(d.referencing_id) AS ref_schema,
       OBJECT_NAME(d.referencing_id) AS referencing_object,
       d.referenced_database_name, d.referenced_schema_name, d.referenced_entity_name
FROM sys.sql_expression_dependencies d
WHERE d.referenced_database_name IS NOT NULL
  AND d.referenced_database_name <> DB_NAME();      -- run per database

-- A3. DYNAMIC SQL census — dependency views cannot see into strings; hunt the text
SELECT OBJECT_SCHEMA_NAME(m.object_id) AS s, OBJECT_NAME(m.object_id) AS o,
       LEN(m.definition) AS len,
       CASE WHEN m.definition LIKE '%sp_executesql%' THEN 1 ELSE 0 END AS uses_sp_executesql,
       CASE WHEN m.definition LIKE '%EXEC%(%'        THEN 1 ELSE 0 END AS uses_exec_string
FROM sys.sql_modules m
WHERE m.definition LIKE '%sp_executesql%' OR m.definition LIKE '%EXEC%(%'
ORDER BY len DESC;                                   -- run per database

-- A4. Multi-part ("dotted") names inside module text — candidates for cross-DB/linked use
SELECT OBJECT_SCHEMA_NAME(m.object_id) AS s, OBJECT_NAME(m.object_id) AS o
FROM sys.sql_modules m
WHERE m.definition LIKE '%].[[]%].[[]%'              -- [x].[y].[z] pattern
   OR m.definition LIKE '%..%';                      -- Db..Table shorthand
-- (Also grep application source for the same patterns — dynamic SQL built in code!)

-- A5. Linked servers & their targets (MI supports SQL-family targets only)
SELECT name, product, provider, data_source, is_linked FROM sys.servers WHERE is_linked = 1;

-- A6. Agent jobs & step types (MI Agent: T-SQL steps OK; CmdExec/PowerShell/proxies NOT)
SELECT j.name, s.step_id, s.subsystem, LEFT(s.command, 200) AS command_preview,
       sch.name AS schedule_name
FROM msdb.dbo.sysjobs j
JOIN msdb.dbo.sysjobsteps s ON s.job_id = j.job_id
LEFT JOIN msdb.dbo.sysjobschedules js ON js.job_id = j.job_id
LEFT JOIN msdb.dbo.sysschedules sch ON sch.schedule_id = js.schedule_id
ORDER BY j.name, s.step_id;

-- A7. Feature red flags for PaaS targets
SELECT name FROM sys.assemblies WHERE is_user_defined = 1;            -- CLR
SELECT * FROM sys.service_queues WHERE is_ms_shipped = 0;             -- Service Broker
EXEC sp_configure 'xp_cmdshell';                                      -- xp_cmdshell enabled?
SELECT instance_name, cntr_value FROM sys.dm_os_performance_counters
WHERE object_name LIKE '%Deprecated Features%' AND cntr_value > 0
ORDER BY cntr_value DESC;                                             -- what old syntax still runs

B. Application source census (PowerShell, run at the solution root)

# B1. In-memory state & context coupling (T3/T6 sizing)
$patterns = 'HttpContext\.Current', 'Session\[', 'Application\[', 'Cache\[',
            'HttpRuntime\.Cache', 'Shared\s+\w+\s+As', 'static\s+\w+'   # VB Shared + C# static
foreach ($p in $patterns) {
  $n = (Get-ChildItem -Recurse -File | Where-Object Extension -in '.vb','.cs','.aspx','.ascx' |
        Select-String -Pattern $p).Count
  '{0,-28} {1,6}' -f $p, $n
}

# B2. Sandbox & environment tripwires (T-sandbox, files, mail, time)
$tripwires = 'System\.IO\.(File|Directory)', 'Server\.MapPath', '[A-Z]:\\\\', '\\\\\\\\\w+\\',  # drive/UNC paths
             'SmtpClient', 'System\.Drawing', 'Microsoft\.Win32\.Registry', 'CreateObject\(',   # GDI/registry/COM
             'DateTime\.Now'                                                                     # local-time logic
foreach ($p in $tripwires) {
  Get-ChildItem -Recurse -File | Where-Object Extension -in '.vb','.cs','.aspx','.config' |
    Select-String -Pattern $p | Group-Object Path | Sort-Object Count -Descending |
    Select-Object -First 5 Name, Count | Format-Table -AutoSize
}

# B3. Dynamic SQL built in application code (A4's blind spot)
Get-ChildItem -Recurse -Include *.vb, *.cs |
  Select-String -Pattern 'ExecuteQuery|ExecuteCommand|SqlCommand|"\s*SELECT|"\s*EXEC' |
  Group-Object Path | Sort-Object Count -Descending | Select-Object -First 20 Name, Count

# B4. Long-request suspects from IIS logs (T6 histogram; adjust log path/fields)
# time-taken is the last field in the default W3C format, in milliseconds
Get-ChildItem 'C:\inetpub\logs\LogFiles\W3SVC1\*.log' | ForEach-Object {
  Get-Content $_ | Where-Object { $_ -notmatch '^#' } | ForEach-Object {
    $f = $_ -split ' '
    [pscustomobject]@{ Url = $f[4]; Ms = [int]$f[-1] }
  }
} | Where-Object Ms -gt 180000 |                                   # > 3 minutes
  Group-Object Url | Sort-Object Count -Descending |
  Select-Object Name, Count, @{n='MaxMin';e={ [math]::Round(($_.Group.Ms | Measure-Object -Maximum).Maximum/60000,1) }}

C. Go/no-go checklist for Option B

  • T6 histogram: every >3-minute page has a conversion plan (or the count forced Option A — also a valid outcome).
  • No COM/GAC/registry-write dependency without a workaround (else: Windows containers / App Service Managed Instance preview / Option A).
  • All linked servers are SQL-family or have replacement plans.
  • All Agent job steps are T-SQL (or moved to WebJobs).
  • MI creation parameters locked: collation + time zone matched to source.
  • <machineKey> pinned; session objects serializable (for the later provider switch).
  • SMTP on 587/ACS; no port-25 senders left.
  • File I/O mapped to %HOME%/Blob; no writes outside the app root.
  • CI produces a deployable package from a clean clone.

Facts current as of mid-2026 and verified against Microsoft documentation: the App Service 230-second (Windows) front-end request limit; Azure SQL Database's lack of cross-database three/four-part-name queries vs Managed Instance's native support; Managed Instance SQL Agent (T-SQL steps, no CmdExec/PowerShell/proxies) and linked servers (SQL-family targets only); Managed Instance Database Mail profile naming; App Service sandbox rules (no registry writes, no out-of-process COM, port 25 blocked) and the preview "Managed Instance on App Service" plan option for COM/GAC/registry-dependent legacy apps. Re-verify limits before committing an architecture — platform constraints evolve.
