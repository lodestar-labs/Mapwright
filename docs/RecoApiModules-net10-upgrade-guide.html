RecoApi Modules → .NET 10 — Modules-Only Upgrade Runbook

Step-by-step guide for upgrading the modules repository alone (Subscribers + Filters) to .NET 10 against an already-migrated host — using Migrate-RecoApiModules.ps1 for the mechanical work and converting each module's AutoMapper maps to Mapperly by hand. Written for a junior engineer; scales from four modules to hundreds.

Read this first — why "modules-only" still means "in lockstep"

The host loads every module DLL with Assembly.LoadFile into its own process. There is no isolation boundary: host and modules share one set of loaded assemblies, so they must agree on the version of everything they share — EF Core, Microsoft.Extensions.*, Newtonsoft.Json, the host's own RECO.API.* DLLs. This is the same-process rule. Consequently a module upgrade is never independent: the day the host goes to .NET 10, every module must be rebuilt on .NET 10 against the new host binaries before the service restarts. "Modules-only" describes which repository you are editing, not a freedom to lag behind. With hundreds of subscribers this lockstep is exactly why the process must be scripted, repeatable and batchable.

1. Module anatomy — what you're touching

Every module is a small class library that plugs into the host. The four files/concepts that matter for the upgrade:

2. Prerequisites

  • The host repo is already migrated (its projects target net10.0) and checked out as a sibling folder of the modules repo, with the expected names — module HintPaths climb ..\..\..\RecoApi\..., so the relative layout is part of the contract.
  • .NET 10 SDK on PATH (dotnet --list-sdks shows a 10.x entry).
  • Clean git tree in the modules repo (git status empty), or plan to use -SkipGit.
  • Migrate-RecoApiModules.ps1 available. If PowerShell refuses to run it: powershell -ExecutionPolicy Bypass -File .\Migrate-RecoApiModules.ps1 ...

Built-in safety net: the lockstep guard

You don't have to remember the first prerequisite — the script refuses to run (exit code 2, nothing changed) if any host project still targets the old TFM, telling you to migrate or pull the host first. Modules compiled against a stale net6 host are the worst failure mode in a plugin system: they build, deploy, load — and then explode at runtime with type-mismatch errors that point nowhere near the real cause.

3. Step 1 — Run the script (Part A, mechanical)

3a. Dry run (default — writes nothing)

PS> .\Migrate-RecoApiModules.ps1 -ModulesRepo C:\src\RecoApiModules -HostRepo C:\src\RecoApi

Read the report: every module should show a WOULD ... (TFM net6.0->net10.0; HintPath ...; packages ...) line or already migrated. Note the line "Derived N package version(s) from host csproj files" — the version targets are read from your actual host, not hard-coded, so the script self-adapts if your host runs different package versions than the reference migration.

3b. Apply, with the build gate

PS> .\Migrate-RecoApiModules.ps1 -ModulesRepo C:\src\RecoApiModules -HostRepo C:\src\RecoApi `
        -Apply -BuildHostFirst

What happens, in order: git branch net10-upgrade created/checked out (modules repo only) → every touched file backed up to a .recobak sibling → csproj transforms → AutoMapper inventory CSV written → host solution built (-BuildHostFirst) → the presence of every host DLL referenced by a module HintPath is verified → each module is built, failures collected rather than aborting (with many modules, one red must not hide the state of the rest) → summary with a per-module OK/FAIL count.

To undo an apply: -Rollback restores every .recobak. Re-running is always safe — the script is idempotent and reports "already migrated" instead of double-editing.

5. Step 2 — AutoMapper → Mapperly, per module (Part B, manual)

Why this step is not optional and not scriptable

Not optional: AutoMapper 11 builds green on .NET 10 but throws TypeInitializationException → VerificationException the moment its MapperConfiguration is constructed (its scanner reflects over LINQ internals that gained struct constraints). Every subscriber using it will load and then die on first mapping. Not scriptable: each ForMember(x => x.Y, o => o.Ignore()) encodes deliberate intent — which key/audit/navigation properties must not be copied onto tracked EF entities. Converting is a transcription of intent, verified by the compiler — a human task with a mechanical rhythm.

Each module needs the same four moves. Budget ~30–60 minutes per typical subscriber once you've done one.

Move 1 — Swap the package

In the module csproj: delete <PackageReference Include="AutoMapper" .../>, add <PackageReference Include="Riok.Mapperly" Version="4.3.1" /> (match the version your host's migration standardised on). Mapperly is a compile-time source generator — no runtime DLL ships, which is precisely why it deletes the version-unification problem AutoMapper had across the plugin boundary.

Move 2 — Transcribe the maps into a [Mapper] class

Create Mapping\SubscriberMappings.cs. Real before/after from a migrated subscriber:

Before — runtime AutoMapper configAfter — compile-time Mapperly
cfg.CreateMap<Reco.TblCode, Sub.TblCode>()
  .ForMember(m => m.TblCodeId,     o => o.Ignore())
  .ForMember(m => m.TblCodeGroupId,o => o.Ignore())
  .ForMember(m => m.TblCodeGroup,  o => o.Ignore())
  .ForMember(m => m.ForeignId,     o => o.Ignore())
  .ForMember(m => m.AddDate,
             o => o.MapFrom(s => s.Created));
// ...
_mapperConfiguration
    .AssertConfigurationIsValid();
 
[Mapper(PropertyNameMappingStrategy =
          PropertyNameMappingStrategy.CaseInsensitive,
        RequiredMappingStrategy =
          RequiredMappingStrategy.Target)]
internal static partial class SubscriberMappings
{
  [MapperIgnoreTarget(nameof(Sub.TblCode.TblCodeId))]
  [MapperIgnoreTarget(nameof(Sub.TblCode.TblCodeGroupId))]
  [MapperIgnoreTarget(nameof(Sub.TblCode.TblCodeGroup))]
  [MapperIgnoreTarget(nameof(Sub.TblCode.ForeignId))]
  [MapProperty(nameof(Reco.TblCode.Created),
               nameof(Sub.TblCode.AddDate))]
  internal static partial void Copy(
      Reco.TblCode source, Sub.TblCode target);
}

The transcription rules, one per old construct:

  • ForMember(…, Ignore()) → one [MapperIgnoreTarget(nameof(...))]. One attribute per old line — no more, no fewer. Behaviour preservation over tidiness.
  • ForMember(…, MapFrom(s => s.X)) (a rename) → [MapProperty(nameof(Src.X), nameof(Dst.Y))].
  • PropertyNameMappingStrategy.CaseInsensitive on the class mirrors AutoMapper's matching (needed because e.g. domain TblCodeID vs model TblCodeId).
  • RequiredMappingStrategy.Target is the compile-time replacement for AssertConfigurationIsValid(): every destination member must be mapped or explicitly ignored — checked on every build instead of in a static constructor at runtime.
  • Subscriber pipelines call Map(source, target) onto tracked local entities — so declare in-place void Copy(source, target) methods only. Don't add create-maps nothing calls.

The clone gotcha (if your module has any same-type map)

A Mapperly same-type map (TblCode Clone(TblCode s)) compiles to return source; — the same reference, not a copy. Anything that snapshots "old values" before an update would silently compare an object with itself. If you need a clone: var t = new TblCode(); SubscriberMappings.CopyOnto(s, t); return t; — new instance plus the generated in-place copy.

Move 3 — Slim the MapperFactory to a registry

The module keeps its MapperFactory but it now just registers the generated methods into the host's mapper seam (defined in RECO.API.Domain, which every module already references). Real after-state:

using RECO.API.Domain.Mapping;

internal static class MapperFactory
{
    public static IObjectMapper CreateMapper()
    {
        return new ObjectMapperRegistry()
            .RegisterUpdate<Reco.TblCode,        Sub.TblCode>(SubscriberMappings.Copy)
            .RegisterUpdate<Reco.TblCodeRel,     Sub.TblCodeRel>(SubscriberMappings.Copy)
            .RegisterUpdate<Reco.TblCodeType,    Sub.TblCodeGroup>(SubscriberMappings.Copy)
            .RegisterUpdate<Reco.TblCodeTypeRel, Sub.TblCodeTypeRel>(SubscriberMappings.Copy);
    }
}

Every registration line points at a method the compiler verified; a missing pair throws an exception naming the exact (source, target) pair. Delete the old IMapperFactory.cs if present.

Move 4 — Swap the consumer type

In SubscriberBase / SubscriberProvider (and any other consumer): using AutoMapper; → using RECO.API.Domain.Mapping; and IMapper → IObjectMapper. The call sites (_mapper.Map(src, dst)) do not change.

Expected diagnostics — how to read Mapperly's output

RMG errors (unmapped required target member): a real gap — add the missing [MapperIgnoreTarget] (if the old config ignored it) or [MapProperty] (if it was a rename). RMG012 warnings on nested navigation members: usually fine — in the subscriber flow records arrive as scalars from the host API and those navigations are null; the warning documents the boundary. Verify that holds for your module, then leave the warning visible rather than suppressing it.

6. Step 3 — Build & verify the deploy-into-host chain

  1. Host first, always. Modules reference host DLLs by HintPath; a missing/stale host build is the root cause of ~90% of module build errors. (-BuildHostFirst does this for you; the script also verifies every referenced DLL exists and names the missing ones.)
  2. Build each module (its own .sln). Watch it deploy into the host: the csproj OutputPath drops the DLLs into RecoApi\src\Service\RECO.API.Service\Modules\Subscribers\<Name>. If that folder didn't update, the host will keep loading the old module — check OutputPath/OutDir survived your edits and that settings.json has CopyToOutputDirectory=PreserveNewest.
  3. Prove the load path. Green compile ≠ loadable plugin. Start the host service and confirm the log lists your module's provider(s), or run the team's module-load harness if one exists (it resolves every exported type of each module DLL against the host binaries — the strongest cheap check).

7. Scaling to hundreds of modules

  • Batch with -ModuleFilter: -ModuleFilter 'RDBES.*', '*Subscriber*', or per-team prefixes. Part A is safe to run repeatedly — already-migrated modules are skipped — so you can sweep the whole repo mechanically on day one, then work Part B batch by batch.
  • Track Part B with the inventory CSV. Pivot it by folder: modules with zero AutoMapper hits are done after Part A; the rest get an owner and the four moves. The build gate's per-module OK/FAIL list is your progress board.
  • Order by risk: convert one representative subscriber first (your team's "reference conversion" — compare against a migrated one throughout), then fan out. Identical subscriber shapes make later conversions largely mechanical.
  • Lockstep the cutover: the host restart onto .NET 10 requires every deployed module rebuilt. Batches can be prepared on branches over weeks, but they merge and deploy together with the host switch. Plan the restart window accordingly.
  • Parity-check anything you're unsure of: map a fully-populated object through the old config (reference AutoMapper 12.0.1 in a throwaway test project — the last freely-usable version that still initialises on .NET 10) and through your Mapperly methods, compare property-by-property. The reference migration proved 236/236 identical this way — the harness is cheap to rebuild and catches transcription slips instantly.

10. Done checklist (per module)

  • csproj: TFM net10.0, tokenised HintPaths, packages = host versions, AutoMapper reference gone, Mapperly added.
  • SubscriberMappings.cs: one attribute per old ForMember; CaseInsensitive + RequiredMappingStrategy.Target; in-place Copy methods only; no aliasing clone.
  • MapperFactory registers every mapped pair; IMapperFactory.cs deleted; consumers on IObjectMapper.
  • Module builds green (warnings reviewed); deploys into the host's Modules folder.
  • Host loads the module; one end-to-end flow verified against pre-migration output.
  • *.recobak, migrate-modules-*.log, automapper-inventory-*.csv deleted before committing; work committed on net10-upgrade.
