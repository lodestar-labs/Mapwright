==============================================================================
 FULL APP MIGRATION PLAN: .NET 6 TO .NET 10
==============================================================================

WHAT THIS IS
------------
A plan to move the whole app from .NET 6 to .NET 10.

The app is one host plus plugin modules. Modules are of two kinds: filters and
subscribers.

The host loads each module into its own process. The host and the modules run
together in one process. They share the same libraries. So:
  - Host and modules must use the same library versions.
  - Host and modules must all move to .NET 10 together.
  - They deploy as one unit.

ORDER OF WORK
-------------
  1. Do the host first.
  2. Then do the filters and subscribers.
  3. Deploy the host and all modules together, in one restart.

Reason: the modules are built against the host and run inside it. They cannot
go before the host. They cannot lag behind it. A module left on the old
version will load and then crash the running app.

TWO KINDS OF WORK
-----------------
  A. Change the target to .NET 10 and update the libraries.
  B. Fix AutoMapper. AutoMapper breaks on .NET 10. See the AutoMapper section.

TIME NOTES
----------
  - Times are a range.
  - "per module" tasks give the time for one module. Multiply by how many you
    have.
  - The local version has few modules. The real app may have many. See the
    section "MORE FILTERS AND SUBSCRIBERS" for the added time per extra module.


==============================================================================
 HOST TASKS (do these first)
==============================================================================

TASK 1 - Get a clean build first.
  WHY:  You need a working start point. If the app is already broken, you
        cannot tell old errors from new ones. Some modules may already fail to
        build against the host (a host model can be missing an interface).
  HOW:  Install the .NET 10 SDK. Build the host and all modules on .NET 6. Fix
        anything already broken. Get everything green.
  TIME: 0.5 to 1 day.

TASK 2 - Change host projects to .NET 10.
  WHY:  .NET 6 is old and not supported. The app must run on .NET 10.
  HOW:  In each host project file, change net6.0 to net10.0. About 11 projects.
  TIME: 0.25 to 0.5 day.

TASK 3 - Update host NuGet packages.
  WHY:  The packages are many versions old. Old versions do not work on
        .NET 10. Packages: Entity Framework, Microsoft.Extensions, Serilog,
        Swagger, test packages, JSON.
  HOW:  Set each package to a .NET 10 version. Write these versions down. The
        modules must use the same ones later.
  TIME: 0.5 to 1 day.

TASK 4 - Fix the email log sink.
  WHY:  The new Serilog email sink removed the old type it used. This is the
        only build error in the host.
  HOW:  Rewrite the two email-sink files to use the new options type. Keep the
        setting names in appsettings the same, so no config files change.
  TIME: 2 to 4 hours.

TASK 5 - Fix the Swagger using line.
  WHY:  The new Swagger moved a namespace. One "using" line no longer works.
  HOW:  Change "using Microsoft.OpenApi.Models;" to "using Microsoft.OpenApi;".
  TIME: 5 to 15 minutes.

TASK 6 - Fix the SQL connection strings.
  WHY:  The new SQL client encrypts connections by default. A dev SQL Server
        with no trusted certificate is refused. This fails at run time, not at
        build time.
  HOW:  Add "TrustServerCertificate=True" to each connection string.
  TIME: 5 to 15 minutes.

TASK 7 - Fix the module folder paths.
  WHY:  The module folder paths are absolute and from the old dev machine. The
        module loader does not accept relative paths.
  HOW:  Put relative paths in appsettings. In the loader, turn them into full
        paths using the app base directory before loading.
  TIME: 15 to 30 minutes.

TASK 8 - Fix AutoMapper in the host.
  WHY:  AutoMapper 11 builds but crashes the first time it maps on .NET 10.
  HOW:  Upgrade AutoMapper, or replace it with Mapperly. See the AutoMapper
        section for both options.
  TIME: Upgrade: 2 to 4 hours. Replace: 1.5 to 2.5 days.


==============================================================================
 MODULE TASKS (do after the host builds; for each filter and subscriber)
==============================================================================

TASK 9 - Change each module to .NET 10.
  WHY:  A module must run on the same runtime as the host, or it will not load.
  HOW:  In the module project file, change net6.0 to net10.0.
  TIME: 5 to 10 minutes per module.

TASK 10 - Match each module's package versions to the host.
  WHY:  Host and module share one process. Different versions of a shared
        library make the module load and then crash.
  HOW:  Set each shared package in the module to the same version the host now
        uses.
  TIME: 10 to 20 minutes per module.

TASK 11 - Fix each module's reference paths to the host DLLs.
  WHY:  The reference paths have net6.0 fixed in them. After the change they
        would point at the old .NET 6 host build.
  HOW:  Replace net6.0 in each path with $(Configuration)\$(TargetFramework).
  TIME: 5 to 15 minutes per module.

TASK 12 - Fix AutoMapper in each subscriber.
  WHY:  Subscribers use AutoMapper. Filters usually do not.
  HOW:  If upgrading AutoMapper: nothing extra here; the version bump in Task 10
        covers it. If replacing with Mapperly: convert the subscriber's maps to
        Mapperly. See the AutoMapper section.
  TIME: Replace: 0.5 to 2 hours per subscriber; first one about 1 day; filters
        0. Upgrade: 0.


==============================================================================
 BUILD, TEST, DEPLOY
==============================================================================

TASK 13 - Build the host, then the modules.
  WHY:  Modules use host DLLs, so the host must build first. A build that
        passes does not prove a module loads.
  HOW:  Build the host. Then build each module. Check each module's files land
        in the host's Modules folder.
  TIME: 0.25 to 0.5 day, plus 5 to 15 minutes per module.

TASK 14 - Test on a machine with the dev database.
  WHY:  The AutoMapper crash, the SQL encryption failure, and module loading
        only show at run time. The app opens the database at start.
  HOW:  Start the app. Check it starts, all modules load, one flow per module
        works, an error email is sent, and the tests pass.
  TIME: 1 to 2 days, plus 15 to 30 minutes per module.

TASK 15 - Deploy all together.
  WHY:  Host and modules run as one unit. A module left on the old version
        crashes the running app.
  HOW:  Deploy the host and all modules at the same time. One restart. Keep the
        old version ready to roll back.
  TIME: 0.5 to 1 day.


==============================================================================
 MORE FILTERS AND SUBSCRIBERS (added time per extra module)
==============================================================================
The local version has few modules. The real app may have many. Each extra
module adds fixed time. The mechanical steps are the same. Only subscribers
add mapping work.

ONE EXTRA FILTER:
  - Change to .NET 10 (Task 9):        5 to 10 min
  - Match packages (Task 10):          10 to 20 min
  - Fix reference paths (Task 11):     5 to 15 min
  - AutoMapper:                        0 (filters have none)
  - Build and deploy check (Task 13):  5 to 15 min
  - Test one flow (Task 14):           15 to 30 min
  TOTAL PER EXTRA FILTER:              about 40 to 90 minutes.

ONE EXTRA SUBSCRIBER:
  - Mechanical steps (Tasks 9 to 11):  20 to 45 min
  - AutoMapper (Task 12):
       Replace path:                   0.5 to 2 hours (first one about 1 day)
       Upgrade path:                   0
  - Build and deploy check (Task 13):  5 to 15 min
  - Test one flow (Task 14):           15 to 30 min
  TOTAL PER EXTRA SUBSCRIBER:
       Replace path:                   about 1.5 to 3.5 hours (after the first)
       Upgrade path:                   about 40 to 90 minutes

NOTE: module work runs in parallel. Many developers can do many modules at the
same time. The host work and the single deploy do not run in parallel.


==============================================================================
 THE AUTOMAPPER ISSUE (two options)
==============================================================================
AutoMapper copies data from one object to another. The app uses AutoMapper 11.

The problem: AutoMapper 11 builds fine on .NET 10 but CRASHES the first time it
maps. This is a hard block. You must do one of two things.


OPTION 1 - UPGRADE AUTOMAPPER
  What: move AutoMapper to a version that works on .NET 10 (version 16).
  Good:
    - Fast. Small change. A version bump and a test.
    - The code stays almost the same.
  Watch out:
    - License. AutoMapper became a paid product in July 2025. Version 15 and up
      need a commercial license. There is a free tier for small companies
      (under about 5 million dollars revenue) and for non-profits. Check if you
      qualify.
    - Host and every module must use the same AutoMapper version.
  Time: 2 to 4 hours for the host. Modules covered by the version bump.


OPTION 2 - REPLACE WITH MAPPERLY
  What: remove AutoMapper. Use Mapperly. Mapperly writes the copy code at build
        time.
  Good:
    - Free. Apache license. No license question.
    - Errors show at build time, not at run time.
    - No shared mapper library. Host and modules no longer need the same mapper
      version. This is good for a plugin app.
  Watch out:
    - More work up front. You convert each map by hand.
    - The clone problem: a same-type copy returns the SAME object, not a new
      copy. This breaks "old value" checks. Fix: make a new object, then copy
      the values into it.
  Time: 1.5 to 2.5 days for the host. 0.5 to 2 hours per subscriber. First
        subscriber about 1 day.


WHICH TO PICK
  - Want it fast, and you qualify for the free tier: upgrade (Option 1).
  - Want no license question and a cleaner plugin app: replace (Option 2).
