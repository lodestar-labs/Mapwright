RecoApi → .NET 10 — Upgrade Runbook

A step-by-step guide for a junior engineer to upgrade a local RecoApi host + its modules from .NET 6 to .NET 10, using the Migrate-RecoApi.ps1 helper for the mechanical work and doing the two bespoke bits by hand — with every test and every likely error and its fix.

Read this first — the one thing that matters

RecoApi is a plugin host: the service loads each Filter/Subscriber module at runtime with Assembly.LoadFile. That means the host and every module run in the same process and must agree on the versions of shared assemblies (EF Core, Microsoft.Extensions.*, Newtonsoft.Json). This is the “same-process rule.” Break it and the app builds fine but throws TypeLoadException / MissingMethodException the moment a module is loaded. The whole migration is designed around keeping those versions identical everywhere — the script enforces it, and you must not undo it by hand.

1. The big picture

Going from .NET 6 to .NET 10 crosses four annual releases. Almost no application logic changes — C# and the base libraries are backward-compatible. What actually breaks is the ecosystem around the code: NuGet packages jump several major versions, a few security defaults flipped, and one old reflection-based library stops working at runtime. The migration is split into two parts so that regressions stay attributable:

PartWhatHow it's done here
Part A
MOSTLY AUTOMATED		 Retarget to net10.0; align every NuGet package; fix HintPaths; harden connection strings; two tiny code edits; one email-sink rewrite. The Migrate-RecoApi.ps1 script does all of it except the email-sink rewrite, which it flags for you (Step 3).
Part B
MANUAL		 Replace AutoMapper (which crashes at runtime on .NET 10 and has a security advisory and a commercial-license change) with Mapperly, a compile-time source generator. Bespoke per subscriber — the script produces an inventory worklist, but a human converts each map and verifies it (Step 5).

The trap that makes Part B non-optional

AutoMapper 11.0.1 builds green on .NET 10 but throws at first use. Its config scanner reflects over LINQ and calls MakeGenericMethod with your entity types; .NET's newer struct-constrained internal overloads reject that:

System.TypeInitializationException
 ---> System.Security.VerificationException: Method System.Linq.Enumerable.MaxInteger:
      type argument 'DatedRecord' violates the constraint of type parameter 'T'.

So a repo that only did Part A will start and then die the moment any mapper is used. You must do Part B before the service is usable. (It doesn't surface in a quick smoke test only because SQL Server connects first.)

The honest boundary

The service opens SQL Server during startup (the persistent queue and the MSSqlServer log sink both create tables). Full runtime verification therefore requires a machine that can reach your dev SQL Server. Everything up to the first DB call — config load, Serilog init, module loading — can be checked anywhere. Plan to do Step 6's runtime items on a DB-connected box.

2. Your repo will differ from the reference — why that's fine

The reference migration was done on a copy with 4 modules. Your production repo may have hundreds of subscribers, maybe an extra authentication layer, or slightly different folder names. That's expected. The script is discovery-based: it globs for every *.csproj and appsettings*.json it can find and transforms each one the same way, so the number of modules does not matter. What must be the same for this runbook to apply (and it is, by design of RecoApi) is the architecture:

✔ Plugin loading via Assembly.LoadFile ✔ Generic Host + Startup bootstrap ✔ Persistent queue on SQL Server ✔ Modules output into the host's Modules\ folders ✔ EF Core + Newtonsoft shared host↔module

If your repo added an authentication layer, it is just more projects/packages — the script will retarget them and align any Microsoft.* packages via its wildcard rules. Only if auth pulls in a third-party package that isn't in the version map will you need to bump it by hand (the build will tell you; see troubleshooting). Nothing about auth changes the steps below.

3. Before you start

  • .NET 10 SDK installed. Verify: dotnet --list-sdks shows a 10.x line.
  • Windows PowerShell 5.1+ (built into Windows) or PowerShell 7. Check: $PSVersionTable.PSVersion.
  • Git, and a clean working tree in both repos (git status shows nothing to commit).
  • Both repos cloned side by side, in the layout the modules expect (module HintPaths point at ..\..\..\RecoApi\..., i.e. the modules repo sits next to the host repo). Keep that layout.
  • Access to the dev SQL Server for the final runtime tests (Step 6). Earlier steps don't need it.
  • Copy Migrate-RecoApi.ps1 somewhere handy (it does not need to live inside either repo).

PowerShell execution policy

If PowerShell refuses to run the script (“running scripts is disabled”), launch it for this one process only:
powershell -ExecutionPolicy Bypass -File .\Migrate-RecoApi.ps1 ...

Step 0 — Map your repository

Spend five minutes locating the pieces so the later errors make sense. In the host repo you should find:

# the host solution and its ~11 projects
RECO.API.sln
src\Service\RECO.API.Service          # the web host — this is what you run
src\Infrastructure\RECO.API.ModulesLoading   # the Assembly.LoadFile plugin loader
src\Infrastructure\RECO.API.PersistentQueue  # the SQL-backed queue
src\Common\RECO.API.Utils\EmailSink   # the custom Serilog email sink (manual rewrite lives here)
src\Service\RECO.API.Service\appsettings.json # connection strings + module folder paths

In the modules repo, each subscriber/filter is its own project that builds into the host's Modules\Subscribers\<Name> or Modules\Filters\<Name> folder (via OutputPath), and references host DLLs by <HintPath>. Confirm you can see many such module folders — that's your per-subscriber Part B workload.

Step 1 — Establish a green .NET 6 baseline

Do not start upgrading on top of a repo that doesn't already build. If it's red on .NET 6, you won't be able to tell upgrade errors from pre-existing ones.

  1. Build the host solution on its current framework: dotnet build RECO.API.sln -c Debug. It must succeed.
  2. Build each module. If a module fails with CS0311 (“the type cannot be used as type parameter…”), that's pre-existing model drift, not the upgrade. The reference repo hit exactly this: a persistence model had lost an interface marker its generic pipeline needs. Fix it the same way every sibling model is written — e.g. public partial class RelType : IDataRecordWithRelationships — then rebuild. Record a green baseline.
  3. Commit the baseline (or at least confirm git status is clean) so the script can create its branch.

Step 2 — Run the script (Part A)

2a. Dry run first — it writes nothing

Dry-run is the default. It prints exactly what it would change, so you can eyeball it before committing.

PS> .\Migrate-RecoApi.ps1 -HostRepo C:\src\RecoApi -ModulesRepo C:\src\RecoApiModules

Read the output. Every project should show either a WOULD ... (TFM net6.0->net10.0; ...) line or already migrated. If a project you expected is missing, check it isn't under a bin\/obj\ folder (those are skipped on purpose).

2b. Apply

When the dry run looks right, add -Apply. The script now: creates/checks out the net10-upgrade branch in each repo, backs up every file it touches to a .recobak sibling, makes the edits, then builds host-then-modules as a gate.

PS> .\Migrate-RecoApi.ps1 -HostRepo C:\src\RecoApi -ModulesRepo C:\src\RecoApiModules -Apply

Expect the build gate to be RED on this first run — that is normal

If your repo has the custom EmailSink (most do), the gate will report the host build FAILED with CS0246: 'EmailConnectionInfo' could not be found — that is precisely the Step 3 work you haven't done yet. Because the Application project depends on Utils, every module will then also fail with a wall of CS0246: 'RECO' could not be found errors. Don't chase those: they are a cascade from the single EmailSink error (the host DLLs the modules reference were never produced). Do Step 3, commit your work on the net10-upgrade branch (the script's preflight requires a clean tree; delete the *.recobak/log/csv artifacts first so they stay out of the commit), then re-run the script — it reports every file "already migrated" and the gate goes GREEN. This exact RED→fix→GREEN sequence was reproduced when validating this runbook.

2c. What the script changed

#ChangeDetail

What the script deliberately does not do

  • The EmailSink rewrite — a real code change; the script flags the files as TODO instead of guessing. → Step 3.
  • Part B (AutoMapper → Mapperly) — it writes an automapper-inventory-*.csv worklist and stops. → Step 5.
  • Pre-existing drift fixes — those are yours to make in Step 1.

Step 3 — Rewrite the custom EmailSink (manual)

MANUAL Skip this step only if your repo has no custom email sink (the script's summary will not list any Serilog.Sinks.Email TODO). Otherwise: Serilog.Sinks.Email 4.x moved from System.Net.Mail to MailKit and deleted EmailConnectionInfo. The sink is bound by name from appsettings.json, so you keep the settings property names and the extension-method name identical — no config file edits — and only rewrite the internals. Here is the exact target:

EmailSinkSettings.cs — the conversion method

// using MailKit.Security;  Serilog.Sinks.Email;  Serilog.Formatting.Display;  System.Net;
public EmailSinkOptions ToEmailSinkOptions()
{
    var result = new EmailSinkOptions
    {
        From = FromEmail,
        To   = new List<string> { ToEmail },
        Host = MailServer,
        Port = Port,
        // EnableSsl=true  → Auto (MailKit negotiates SSL/STARTTLS from the port)
        // EnableSsl=false → None (plain SMTP, matching the old default on port 25)
        ConnectionSecurity = EnableSsl ? SecureSocketOptions.Auto : SecureSocketOptions.None,
        IsBodyHtml = IsBodyHtml,
        // 4.x subjects are Serilog message templates, not plain strings:
        Subject = new MessageTemplateTextFormatter(EmailSubject ?? "Log Email"),
    };
    // old sink only set credentials when UseDefaultCredentials was false — preserve that:
    if (!UseDefaultCredentials)
        result.Credentials = new NetworkCredential(Username, Password);
    return result;
}

EmailSinkExtensions.cs — the registration (name + signature unchanged)

public static LoggerConfiguration EmailSink(this LoggerSinkConfiguration loggerConfiguration,
                                            EmailSinkSettings emailSinkSettings)
{
    if (emailSinkSettings == null) throw new ArgumentNullException(nameof(emailSinkSettings));
    var options = emailSinkSettings.ToEmailSinkOptions();
    options.ServerCertificateValidationCallback = ValidateServerCertificate;
    return loggerConfiguration.Email(options, restrictedToMinimumLevel: emailSinkSettings.MinimumLogLevel);
}

private static bool ValidateServerCertificate(object sender, X509Certificate certificate,
                                              X509Chain chain, SslPolicyErrors sslPolicyErrors) => true;

This is the only compile error you should see in the host after Part A. Once it's in, the host compiles. (You still have Part B before it runs.)

Step 4 — Build in the right order

Order matters, and “stop the app first”

Modules reference the host's DLLs by HintPath, so build the host first, then the modules. And because a running host locks RECO.API.*.dll, a rebuild will fail with MSB3021/MSB3027 file-lock errors — those are not compile errors. Stop the service before rebuilding.

# stop any running host (Windows)
PS> taskkill /F /IM RECO.API.Service.exe   # name may differ; check Task Manager

# 1) host
PS> dotnet build C:\src\RecoApi\RECO.API.sln -c Debug

# 2) each module (its own .sln, or the whole modules folder)
PS> dotnet build C:\src\RecoApiModules\Subscribers\RECO.Subscriber\RECO.Subscriber.sln -c Debug

The -Apply run already did this build once as a gate and reported GREEN/RED. Re-run manually whenever you change code (Steps 3 and 5). A green build here means Part A is mechanically complete.

Step 5 — AutoMapper → Mapperly (Part B, manual, per subscriber)

MANUAL Open the automapper-inventory-*.csv the script wrote (in the host repo root). Every row is a file + line using AutoMapper. Group them by project — each project with maps needs the same three moves. Why this can't be a blind find-replace: every ForMember(…Ignore()) encodes a deliberate behaviour (which audit columns and navigations must not be copied), and AutoMapper matched property names case-insensitively. Mapperly makes all of that explicit and compile-checked — which is the whole point, but it means a human maps intent, not text.

The three moves per project

  1. Swap the package. In the project's csproj, remove <PackageReference Include="AutoMapper" …/> and add <PackageReference Include="Riok.Mapperly" Version="4.3.1" />. You add this line by hand — the script never adds package references; its version map merely pins Mapperly to 4.3.1 wherever a reference already exists.
  2. Convert each CreateMap to a Mapperly partial method in a [Mapper] class, turning every ForMember(x => x.Y, o => o.Ignore()) into [MapperIgnoreTarget(nameof(Target.Y))] and every ForMember(… MapFrom …) rename into [MapProperty(nameof(Src.A), nameof(Target.B))].
  3. Point consumers at the generated methods. Change the injected type from AutoMapper's IMapper to the project's mapper seam (IObjectMapper); the call sites (_mapper.Map(src, dst)) stay the same.

Worked example — a real subscriber, before → after

Before (AutoMapper, runtime config) After (Mapperly, compile-time) 
cfg.CreateMap<Reco.TblCode, Sub.TblCode>()
  .ForMember(m => m.TblCodeId,     o => o.Ignore())
  .ForMember(m => m.TblCodeGroupId,o => o.Ignore())
  .ForMember(m => m.TblCodeGroup,  o => o.Ignore())
  .ForMember(m => m.ForeignId,     o => o.Ignore())
  .ForMember(m => m.AddDate,
             o => o.MapFrom(s => s.Created));
// ...
_mapperConfiguration.AssertConfigurationIsValid();
 
[Mapper(PropertyNameMappingStrategy =
          PropertyNameMappingStrategy.CaseInsensitive,
        RequiredMappingStrategy =
          RequiredMappingStrategy.Target)]
internal static partial class SubscriberMappings
{
  [MapperIgnoreTarget(nameof(Sub.TblCode.TblCodeId))]
  [MapperIgnoreTarget(nameof(Sub.TblCode.TblCodeGroupId))]
  [MapperIgnoreTarget(nameof(Sub.TblCode.TblCodeGroup))]
  [MapperIgnoreTarget(nameof(Sub.TblCode.ForeignId))]
  [MapProperty(nameof(Reco.TblCode.Created),
               nameof(Sub.TblCode.AddDate))]
  internal static partial void Copy(
        Reco.TblCode source, Sub.TblCode target);
}

Note the two attributes on the [Mapper]: CaseInsensitive reproduces AutoMapper's name matching (domain TblCodeID vs model TblCodeId); RequiredMappingStrategy.Target reproduces AssertConfigurationIsValid() — but at build time, on every compile, instead of in a static constructor at runtime.

The one gotcha that will bite you: clone must not alias

If you declare a same-type map (e.g. TblCode Clone(TblCode s)), Mapperly compiles it to return source; — the same reference, not a copy. The repository snapshots “old content” before updating a tracked entity; an aliased snapshot makes every old/new comparison silently equal and corrupts change detection. Never use a Mapperly same-type map for cloning. Instead generate an in-place copy and wrap it:

internal static partial void CopyOnto(TblCode source, TblCode target);   // generated, correct
// clone = new instance + CopyOnto, by hand:
var target = new TblCode(); RecoMappings.CopyOnto(source, target); return target;

Expect (and accept) some RMG warnings

Mapperly may emit RMG012 “unmapped member” warnings for nested navigations it can't match (e.g. AddDate, Db on a nested nav). In the subscriber flow those navigations arrive null (records come as scalars from the host API), so behaviour is unchanged — the warnings just document the boundary. A hard RMG error (an unmapped required target) is different: add the missing [MapperIgnoreTarget] or [MapProperty]. Do a subscriber, build it, read the diagnostics, fix, repeat.

Prove parity, don't assume it

For a subscriber whose maps you're unsure about, populate an object, run it through both the old AutoMapper config and your Mapperly version, and compare every property. The reference migration did exactly this across all entities and got 236/236 identical — and that harness is what caught both the runtime crash and the clone gotcha. At minimum, eyeball that each old ForMember(Ignore) has a matching [MapperIgnoreTarget] before moving on.

Step 6 — The test ladder

Run these in order. Each rung tells you a specific thing works; if one fails, jump to troubleshooting before climbing higher.

#TestCommand / actionExpected — and what it proves
1Host compiles dotnet build RECO.API.sln -c Debug Build succeeded. Part A + EmailSink are correct. (Ignore MSB302x file-locks — stop the app.)
2Every module compiles & deploys build each module .sln Green, and DLLs appear under the host's Modules\…\<Name>. Proves HintPaths + versions line up.
3No AutoMapper left re-run the script (dry-run) or grep the tree for AutoMapper Inventory reports 0 references. Proves Part B is complete.
4Unit tests dotnet test on the test project(s) Compile & run. NUnit 3→4 asserts still work. (Integration tests needing SQL will fail only for lack of DB — not a regression; see below.)
5Service starts & loads modules DB dotnet run --project src\Service\RECO.API.Service Starts without the AutoMapper crash; log shows each module loaded; Swagger UI answers; the MSSqlServer log table is created. Proves the plugin path + mapping + Serilog stack on .NET 10.
6One end-to-end flow per module DB trigger a subscriber pull → mapping → persist Data lands in SQL with correct values (esp. renamed columns like AddDate). Proves Mapperly parity for real.
7Error email DB force a failing subscriber An email is sent via the rewritten sink. This is the one config-shape change worth an explicit test.
8Persistent-queue tests DB dotnet test (queue tests) Green against the dev DB. Proves EF Core 6→10 query behaviour on the queue.

Interpreting “2 tests failed”

If the only failures are integration tests that need SQL Server and you're on a machine that can't reach it, that is not an upgrade regression — it's an environment gap. Re-run them on the DB-connected box (rung 8). Confirm the failure message is a connection/login error, not an assertion or type error.

Error navigation — troubleshooting

Find your symptom. The Class column tells you when it appears: BUILD at compile, RUN only at runtime.

Build-time

SymptomClassCause & fix
MSB3021 / MSB3027: cannot copy/write RECO.API.*.dll, file in use BUILD Not a real error. The host is running and holding the DLL. Stop it (taskkill /F /IM RECO.API.Service.exe) and rebuild.
CS0246: Microsoft.OpenApi.Models not found BUILD Swashbuckle 10 flattened the namespace. Change the using to Microsoft.OpenApi; (the script does this — this only appears if a file was missed).
CS0246: EmailConnectionInfo not found BUILD Sinks.Email 4.x removed it. Do the Step 3 rewrite.
CS0311: type can't be used as generic parameter (…IDataRecordWithRelationships) BUILD Pre-existing model drift, not the upgrade. Add the missing interface marker to the model, as the sibling models have it (Step 1).
NU1102/NU1202: package X version Y not compatible with net10.0 BUILD A package (often from an auth layer) isn't in the version map. Find its net10-compatible version on nuget.org and either bump it in the csproj or add it to a -ConfigFile JSON and re-run.
NU1605: detected package downgrade BUILD A module pins a shared package lower than the host (or vice-versa). Make them equal — that's the same-process rule. Align to the version map value.
MSB3245 / CS0246 on a host type inside a module; “reference could not be resolved” BUILD Either the host wasn't built first, or a stale bin\Debug\net6.0 HintPath is still pointing at a deleted folder. Confirm the HintPath uses $(Configuration)\$(TargetFramework) and rebuild the host, then the module. dotnet clean if a stale net6 bin lingers.
NU1903: known high-severity vulnerability in AutoMapper BUILD (warning) Expected until Part B removes AutoMapper. It's a signal, not a blocker.
RMG… Mapperly error: target member not mapped BUILD A required destination property has no source and no ignore. Add [MapperIgnoreTarget(nameof(...))] (if it should stay unset — e.g. audit/nav columns) or [MapProperty(...)] (if it's a rename).
RMG012 warning: unmapped member on a nested navigation BUILD (warning) Usually fine — the nav is null in the subscriber flow. Verify that's true for your case, then leave it; it documents the boundary (Step 5).

Rollback & safety net

You have two independent undo levels — use whichever is handier:

  1. Script backups. Every -Apply run copies each file to <file>.recobak before editing. To undo the last apply: 
    PS> .\Migrate-RecoApi.ps1 -HostRepo C:\src\RecoApi -ModulesRepo C:\src\RecoApiModules -Rollback
    It restores every backup and deletes the .recobak files. (Any git branch it created is left in place — delete it by hand if you want.)
  2. Git. Because the script works on the net10-upgrade branch, you can always git checkout . to discard working changes, or git checkout main and git branch -D net10-upgrade to throw the whole attempt away. Nothing is pushed until you push it.

Finish & merge checklist

  • Host builds green on net10; all modules build green and deploy into Modules\.
  • EmailSink rewritten; no EmailConnectionInfo left.
  • Inventory shows 0 AutoMapper references; Mapperly builds with no RMG errors.
  • Clone/copy semantics verified (no identity-alias); mapping parity spot-checked.
  • Service starts, loads every module, Swagger answers DB.
  • One end-to-end flow per module passes; error email fires; queue tests green DB.
  • Before committing, delete the script's working artifacts so they don't land in history: every *.recobak backup (only once you're sure you won't roll back), plus the migrate-recoapi-*.log and automapper-inventory-*.csv files in the host repo root.
  • Commit Part A and Part B as separate commits/branches so regressions stay attributable. Merge order: net10 retarget first, then the Mapperly change.
  • Update your team's connection strings on each environment (they now carry TrustServerCertificate=True — keep it unless the server has a trusted cert).

C. What the script is not allowed to decide for you

By design it never: rewrites the EmailSink, converts AutoMapper maps, fixes model drift, pushes to a remote, or edits anything under bin\/obj\. Those either need human judgement or aren't source. Everything it does touch is backed up and reversible.

Runbook companion to the executed migration report (net10-migration-report.html). The mechanical transforms in Migrate-RecoApi.ps1 were validated by reproducing the reference migration from a clean .NET 6 checkout and diffing byte-for-byte against the known-good result: modules identical, host identical apart from two comment lines and one extra (beneficial) hardened connection string. Part B remains a human task by design.
