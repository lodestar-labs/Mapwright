The shape of this project: unlike the host migration (one deep, sequential effort), the module fleet is a wide, shallow, parallelisable work stream. The expensive thinking has already been done once on the reference modules; what remains per module is (a) a scripted sweep that costs minutes per hundred modules, and (b) a patterned, compiler-checked manual conversion of each module's data-mapping configuration, which distributes across as many developers as we choose to assign. The critical management constraint is not effort but coordination: because all modules load into the host's single process, the host switch and every module's rebuilt binaries must deploy in one restart window — modules cannot lag.

  • Recommended model: distribute conversion to the teams that own each subscriber (they know their maps' intent — the one thing the work actually requires), coordinated by one rollout owner who runs the script sweeps, keeps the tracking sheet, and audits a sample of conversions. Elapsed time then depends on scheduling, not on a single person's throughput.
  • Not every module pays the human cost: modules with zero mapping-library usage (the CSV shows this immediately — e.g. the sample filter in the reference estate) are done after the scripted sweep. Expect the real fleet to bifurcate; the estimate above conservatively prices every module.
  • Prioritisation: convert one representative subscriber per team first (the pattern anchor), then fan out alphabetically — variance comes almost entirely from map count, which the CSV makes visible up front.

5. The coordination constraint (management's real job here)

One process, one restart window

Every module loads into the host's process; host and modules must agree on all shared library versions. Therefore branches can be prepared over weeks, in parallel — but the host switch and all rebuilt modules deploy together, once. A single unconverted module that still constructs the old mapping library will load and then crash the shared service at first use. Mitigations already built in: the script's lockstep guard (refuses to run against an unmigrated host), the build gate's per-module ledger, a pre-deploy grep gate ("zero mapping-library references outside the inventory CSV"), and a module-load verification step in the runbook. The management ask is simply to own the calendar: one agreed freeze + restart window, with the tracking sheet at 100% as its entry criterion.

7. Decisions requested from management

  1. Confirm the staffing model — distributed-to-owners with one rollout coordinator (recommended), or a dedicated squad (§4 table gives the trade-off).
  2. Approve the day-one scripted sweep of the whole fleet on a branch (hours, reversible, produces the exact per-module workload numbers that replace §4's estimates with measurements).
  3. Set the cutover window and adopt "tracking sheet at 100% + grep gate green + load check green" as its entry criteria.
