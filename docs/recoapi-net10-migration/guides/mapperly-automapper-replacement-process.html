Replacing AutoMapper with Mapperly

The exact process, and the real challenges, for swapping a runtime-configured object mapper (AutoMapper) for a compile-time source-generated one (Riok.Mapperly). Every code sample below is drawn from an actual migration, not a synthetic example.

Facts as of this writing

Mapperly latest stable is 4.3.1, license Apache-2.0 (no commercial tier, no revenue threshold — unlike AutoMapper since its July 2025 licensing change). Mapperly ships zero runtime library: it is a Roslyn source generator that writes ordinary C# into your assembly at build time. That one fact is the root of both its biggest benefit and its biggest process cost, explained below.

1. Why replace AutoMapper at all

Three independent reasons converged, but only the first is a hard blocker:

ReasonDetailForces replacement?
Runtime crash on newer .NET Older pinned AutoMapper versions (e.g. 11.x) build cleanly but throw VerificationException the first time a mapper configuration is constructed, because their config scanner reflects over LINQ internals that gained stricter generic constraints in newer runtimes. YES for that pinned version — but see the note below.
Security advisory on the old pinned version A high-severity advisory is flagged by NuGet audit on every build until the version changes. No — an upgrade also clears this.
Commercial licensing (as of July 2025) AutoMapper v15+ is dual-licensed (RPL-1.5 / commercial); a free community tier exists for organisations under roughly $5M revenue and for non-profits. No — check if you qualify for the free tier first.

The honest fork in the road

Newer AutoMapper releases (v16+) do support current .NET runtimes. So the crash alone only forces an AutoMapper version bump, not a switch to Mapperly. Replacing with Mapperly is a deliberate choice for the reasons in the next sections — mainly the licensing question and, for multi-assembly plugin systems, the version-coupling problem Mapperly eliminates. If neither of those apply to you, upgrading AutoMapper in place is faster. This document is written for the case where you decide replacement is worth it.

2. The exact process, step by step

This is the sequence actually followed, per mapping-bearing project (a host data layer or a plugin module), with real before/after code.

Step 1 — Swap the package reference

- <PackageReference Include="AutoMapper" Version="11.0.1" />
+ <PackageReference Include="Riok.Mapperly" Version="4.3.1" />

Mapperly is a build-time analyzer package (PrivateAssets="all" by default in its own package metadata) — it does not appear in your published output as a runtime dependency.

Step 2 — Design the seam (do this once, before converting any map)

This is the step that is easy to skip and shouldn't be. AutoMapper's IMapper exposes a generic, runtime-queryable API — any caller can write _mapper.Map<TDestination>(source) for any registered pair, resolved at runtime. Mapperly generates the opposite: concrete, statically-typed partial methods, one per mapping, known at compile time. If your codebase has generic consumers (a repository class templated on the entity type, for example), you need a small bridge between the two shapes.

// A tiny seam, once, in a shared assembly:
public interface IObjectMapper
{
    TDestination Map<TSource, TDestination>(TSource source);
    void Map<TSource, TDestination>(TSource source, TDestination destination);
}

public sealed class ObjectMapperRegistry : IObjectMapper
{
    private readonly Dictionary<(Type, Type), Delegate> _updates = new();
    private readonly Dictionary<(Type, Type), Delegate> _creates = new();

    public ObjectMapperRegistry RegisterUpdate<TSource, TDestination>(
        Action<TSource, TDestination> map)
    {
        _updates[(typeof(TSource), typeof(TDestination))] = map;
        return this;
    }
    // ...Register<TSource,TDestination>(Func<TSource,TDestination> create) analogous...

    public void Map<TSource, TDestination>(TSource source, TDestination destination)
    {
        if (!_updates.TryGetValue((typeof(TSource), typeof(TDestination)), out var d))
            throw new InvalidOperationException(
                $"No mapping registered for ({typeof(TSource).Name}, {typeof(TDestination).Name}).");
        ((Action<TSource, TDestination>)d)(source, destination);
    }
}

Every registration line points at a Mapperly-generated method the compiler already verified; a missing pair throws an exception naming the exact pair at first use, instead of AutoMapper's old build-time AssertConfigurationIsValid(). Host it in whichever assembly all your mapping-bearing projects already reference (a shared domain/abstractions project in a plugin system).

Step 3 — Convert each CreateMap into a Mapperly [Mapper] partial method

Real before/after, from a subscriber module that copies a code-reference record into a local repository model:

Before — AutoMapper, runtime-configuredAfter — Mapperly, compile-time
cfg.CreateMap<Reco.TblCode, Sub.TblCode>()
  .ForMember(m => m.TblCodeId,     o => o.Ignore())
  .ForMember(m => m.TblCodeGroupId,o => o.Ignore())
  .ForMember(m => m.TblCodeGroup,  o => o.Ignore())
  .ForMember(m => m.ForeignId,     o => o.Ignore())
  .ForMember(m => m.AddDate,
             o => o.MapFrom(s => s.Created));
 
[Mapper(PropertyNameMappingStrategy =
          PropertyNameMappingStrategy.CaseInsensitive,
        RequiredMappingStrategy =
          RequiredMappingStrategy.Target)]
internal static partial class SubscriberMappings
{
  [MapperIgnoreTarget(nameof(Sub.TblCode.TblCodeId))]
  [MapperIgnoreTarget(nameof(Sub.TblCode.TblCodeGroupId))]
  [MapperIgnoreTarget(nameof(Sub.TblCode.TblCodeGroup))]
  [MapperIgnoreTarget(nameof(Sub.TblCode.ForeignId))]
  [MapProperty(nameof(Reco.TblCode.Created),
               nameof(Sub.TblCode.AddDate))]
  internal static partial void Copy(
        Reco.TblCode source, Sub.TblCode target);
}

The transcription rules — apply mechanically, one old line at a time:

  • .ForMember(x => x.Y, o => o.Ignore()) → one [MapperIgnoreTarget(nameof(Target.Y))]. One attribute per old ignore line, no more, no fewer.
  • .ForMember(x => x.Y, o => o.MapFrom(s => s.Z)) (a rename) → [MapProperty(nameof(Source.Z), nameof(Target.Y))].
  • PropertyNameMappingStrategy.CaseInsensitive on the [Mapper] class reproduces AutoMapper's default case-insensitive property matching (needed wherever, e.g., a domain property is TblCodeID and the persistence model is TblCodeId).
  • RequiredMappingStrategy.Target reproduces AssertConfigurationIsValid() — every destination member must be mapped or explicitly ignored — but the check now runs on every build, not inside a static constructor the first time the app starts.

Step 4 — Register the generated method, matching what the pipeline actually calls

internal static class MapperFactory
{
    public static IObjectMapper CreateMapper()
    {
        return new ObjectMapperRegistry()
            .RegisterUpdate<Reco.TblCode,        Sub.TblCode>(SubscriberMappings.Copy)
            .RegisterUpdate<Reco.TblCodeRel,     Sub.TblCodeRel>(SubscriberMappings.Copy)
            .RegisterUpdate<Reco.TblCodeType,    Sub.TblCodeGroup>(SubscriberMappings.Copy)
            .RegisterUpdate<Reco.TblCodeTypeRel, Sub.TblCodeTypeRel>(SubscriberMappings.Copy);
    }
}

Grep the pipeline code first: register only the (source, target) pairs something actually calls. Registering unused create-maps just because the old config had them adds nothing but noise.

Step 5 — Swap the consumer type

- using AutoMapper;
+ using RECO.API.Domain.Mapping;
...
- protected internal readonly IMapper _mapper;
+ protected internal readonly IObjectMapper _mapper;
- internal SubscriberBase(ILogger logger, IContextFactory f, IMapper mapper)
+ internal SubscriberBase(ILogger logger, IContextFactory f, IObjectMapper mapper)

Call sites do not change — _mapper.Map<T>(x) and _mapper.Map(src, dst) read identically before and after. Only the injected type changes, which is why this step is fast even across many consumer classes.

Step 6 — Handle same-type copies specially (see the clone challenge below)

// Never a Mapperly same-type map. Always: new instance + generated in-place copy.
.Register<Models.TblCode, Models.TblCode>(source =>
    { var target = new Models.TblCode(); RecoMappings.CopyOnto(source, target); return target; })

Step 7 — Prove equivalence, don't assume it

Build a small harness that maps a fully-populated object through the original AutoMapper configuration and through the new Mapperly-backed registry, then compares every property. In the reference migration this harness compared 236 property values across 8 entity types × 4 map shapes (to-model, to-domain, clone, copy-onto) and found them 236/236 identical — and it is also what caught the clone bug below, before it ever reached a test environment.

3. The real challenges

Challenge 1 — The identity-clone trap SILENT DATA CORRUPTION

The single most dangerous thing encountered doing this. Ask Mapperly for a same-type clone — [MapperIgnoreTarget]-free, straightforward-looking —

internal static partial Models.TblSynonym? Clone(Models.TblSynonym? source);

and it generates, entirely reasonably from its own point of view:

internal static partial Models.TblSynonym? Clone(Models.TblSynonym? source)
{
    return source == null ? default : source;   // ← the SAME reference, not a copy!
}

Mapperly treats "map a type to itself" as an identity — logically correct, since every member already matches its own name and type. But if your code snapshots "old values" before an update to compare against "new values" later (a common pattern in a repository layer that logs/validates changes), an aliased snapshot makes every old/new comparison silently equal. No exception. Just wrong behaviour. Fix: never declare a same-type map for cloning — always new instance plus the generated in-place copy method (Step 6 above).

Challenge 2 — Bridging generic consumers to static generated methods DESIGN WORK, NOT MECHANICAL

Covered as Step 2, but worth calling out as a genuine challenge, not just a step: this is the one part of the whole process that is design, not transcription. AutoMapper's runtime type resolution let generic, templated consumer code (e.g. Repository<TDbModel>) call _mapper.Map<TDbModel>(obj) for any of dozens of entity types through one interface. Mapperly's compile-time methods have no equivalent generic entry point by design — that's precisely what makes them reflection-free and fast. Bridging the two shapes (the IObjectMapper + registry seam) has to be designed once, correctly, before mechanical conversion can even begin. Skipping this step and converting maps ad hoc leads to inconsistent call patterns across the codebase.

Challenge 3 — Reproducing two implicit AutoMapper behaviours EASY TO MISS

AutoMapper's defaults are implicit; Mapperly's are explicit attributes you must remember to add on every [Mapper] class:

  • Case-insensitive property matching — without PropertyNameMappingStrategy.CaseInsensitive, Mapperly's default exact-name matching will silently leave case-mismatched properties unmapped (caught, loudly, only if RequiredMappingStrategy.Target is also set — see next point).
  • Configuration completeness checking — AssertConfigurationIsValid() has no automatic Mapperly equivalent; you must opt in with RequiredMappingStrategy.Target on every mapper class, or an unmapped destination member fails silently instead of becoming a compile error.

Both are one-line fixes once known — the challenge is that nothing warns you if you forget them; a mapper class without these two settings compiles and looks correct.

Challenge 4 — Diagnostics on nested/navigation members USUALLY BENIGN, MUST BE VERIFIED

Mapperly auto-derives nested object maps when it can, and emits an RMG012 warning when it can't fully match a nested type's members (common on EF navigation properties like a code's parent group/category, which Mapperly cannot know are irrelevant in a given flow). In the reference migration these warnings were accepted, not suppressed — verified case by case that the navigation in question arrives null in that code path (records come from the host API as scalars, not populated graphs), so the warning correctly documents a boundary rather than hiding a real bug. Treat every RMG012 as a question to answer, not noise to silence.

Challenge 5 — It's a manual, per-mapping transcription job EFFORT, NOT RISK

Nothing above the package swap is automatable safely. Every ForMember(...Ignore()) encodes a deliberate business decision (which audit columns, which navigation properties must never be touched by this specific pipeline) — a blind text-replace risks silently dropping or keeping the wrong ignores. In a plugin system with many subscriber/filter modules, this means the conversion is genuinely proportional to the number of mapping-bearing projects: expect roughly 0.5–2 hours per module once the pattern is set, with the first module taking closer to a day while the seam and the team's own conventions get established.

4. What turned out NOT to be a challenge

  • Call-site churn. Because the seam preserves Map<T>(x) / Map(src, dst) signatures, consumer code outside the mapping definitions themselves barely changes — one interface-type swap per class, nothing else.
  • The plugin version-coupling problem — it actually gets solved, not just avoided. AutoMapper's runtime library had to be the same version across a host and every module sharing its process (a real constraint in Assembly.LoadFile-style plugin architectures). Because Mapperly ships no runtime library — the generated code compiles directly into each assembly — that whole class of constraint disappears once the switch is made. This is a strict win specific to multi-assembly/plugin systems, not a challenge.
  • Build-time performance. Source generation added no perceptible build-time cost in this migration; Mapperly's generator is intentionally lightweight.
  • Test framework compatibility, dependency injection registration style, JSON serialization of mapped objects — all unaffected; the generated methods are ordinary static C#.

6. Verdict — when this trade is worth it

Worth it when: you run a multi-assembly / plugin architecture where a shared runtime mapper version is an operational liability; you want mapping errors caught by the compiler instead of discovered in production; or the AutoMapper licensing change is a real constraint for your organisation.

Not worth it (upgrade AutoMapper instead) when: you have a single deployable, you qualify for AutoMapper's free tier, and speed matters more than the long-term properties above — an AutoMapper version bump is a matter of hours, this process is a matter of days plus a per-module tail.
