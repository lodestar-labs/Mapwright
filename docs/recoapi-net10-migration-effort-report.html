What happened: the full host + modules code base was migrated on branches (net10-upgrade, then replace-automapper-mapperly in both repositories), verified as far as this environment allows, and — more importantly for cost — the migration itself was converted into reusable, tested tooling and runbooks, so re-applying it to the production repositories (which differ mainly in module count) is a fraction of the original effort.

Why it was necessary (not optional):

  • Unsupported runtime. .NET 6 left Microsoft support in November 2024; every month since is unpatched exposure on an internet-adjacent data service.
  • A guaranteed production crash was latent. AutoMapper 11 — used by the host and three subscriber modules — cannot initialise on .NET 10: builds succeed, then the service throws on first mapper use. A "just retarget" migration would have shipped a time bomb. The library also carries a high-severity security advisory, and its fixed versions moved to a commercial licence — so replacement (with Apache-licensed, compile-time Mapperly) was the only path that resolved crash, CVE and licensing at once.
  • Pre-existing drift was already breaking module builds (two of four modules didn't compile against the current host before we started) — evidence that the estate needed a consolidation pass regardless.

4. Quality evidence — why we trust this migration

  • Behaviour preservation was measured, not assumed: the new mapping layer was compared against the original AutoMapper configuration on fully-populated objects — 236 of 236 property comparisons identical across all 8 entities and every map shape (including the subtle clone semantics where the naive conversion would have silently corrupted change detection — caught precisely by this harness).
  • The plugin risk was tested directly: a load harness resolved every exported type of every module (44–66 types each) against the .NET 10 host binaries — the exact failure mode a plugin architecture fears most.
  • The automation was validated like a product: run against clean checkouts of the pre-migration state, its output diffed byte-for-byte against the hand-made reference branches (modules: identical; host: identical except comments), plus idempotency, rollback, and a full failure-path rehearsal (expected RED build → fix → GREEN) so the runbook's troubleshooting section reflects observed reality.
  • Honest boundary: everything above ran without the dev SQL Server. Startup opens the database, so final end-to-end confirmation is an explicit open item (§2), not an implied claim.

6. Decisions requested from management

  1. Schedule the DB-connected verification (1–2 days, any engineer + the runbook checklist) and treat its completion as the merge gate.
  2. Approve the merge order: net10-upgrade first, then replace-automapper-mapperly — two clean diffs, regressions stay attributable. Nothing has been pushed without review.
  3. Commission the production-fleet rollout per the modules-only effort report (staffing model and timeline scenarios there), including one agreed service-restart window for the lockstep cutover.
  4. Adopt the tooling as the standard path for any sibling deployments of this architecture, so the one-off investment keeps paying out.
