RecoApi → .NET 10: The Key Challenges

A developer-to-developer walkthrough of what actually made this migration hard — host + modules combined, and the modules-only variant. Not a runbook (those exist separately): this is the why behind each decision, with the evidence, for anyone facing a similar .NET 6 → 10 upgrade of a plugin-architecture system.

The system in one paragraph

RecoApi is a .NET web service (Generic Host + Startup, controllers, Swagger, Serilog, EF Core over SQL Server, a persistent queue) whose business capabilities live in plugin modules — subscriber and filter DLLs the host discovers at runtime with Assembly.LoadFile from Modules\ folders. Modules live in a second repository, reference the host's DLLs by HintPath, and build directly into the host's module folders. Everything ran on .NET 6 with AutoMapper 11 for entity mapping. The task: move host and modules to .NET 10 — behaviour-preserving, with a reviewable diff.

The challenges, one by one

1. Green build ≠ working system: the AutoMapper time bomb

Applies to: combined & modules-only · Detected by: a parity harness, not by any build

What made it hard

Retargeting all 15 projects to net10.0 compiled almost immediately. Nothing in the build hinted that AutoMapper 11.0.1 cannot even construct its configuration on .NET 10. Its config scanner reflects over LINQ's internal method table and calls MakeGenericMethod with mapped entity types; newer runtimes added internal overloads with struct constraints (Enumerable.MaxInteger<T>), and the reflection call now throws:

System.TypeInitializationException
 ---> System.Security.VerificationException: Method System.Linq.Enumerable.MaxInteger:
      type argument 'DatedRecord' violates the constraint of type parameter 'T'.

Worse: the standard smoke test (start the service) never reached it, because SQL Server connects during startup — the crash hides behind an unrelated infrastructure dependency. Compounding factors: the pinned version carries a high-severity advisory (GHSA-rvv3-g6hj-g44x, flagged by NuGet audit on every build), and fixed versions moved to a commercial licence. Upgrade wasn't a fix; replacement was.

How it was resolved

Replace AutoMapper with Mapperly (Riok.Mapperly, Apache-2.0), a compile-time source generator — but as a separate branch/commit after the retarget, never mixed into it. Every ForMember(...Ignore()) became one [MapperIgnoreTarget]; CaseInsensitive matching and RequiredMappingStrategy.Target reproduced AutoMapper's name-matching and its AssertConfigurationIsValid() — but enforced by the compiler on every build. A small seam (IObjectMapper + ObjectMapperRegistry in the Domain assembly) bridged generic consumers to the generated typed methods, so call sites (_mapper.Map(src, dst)) didn't change.

Evidence

A parity harness mapped fully-populated objects through the original AutoMapper configuration (reference: 12.0.1, the last freely-usable version that initialises on net10) and the Mapperly replacement: 236/236 property comparisons identical across 8 entities × 4 map shapes. The same harness is what exposed the crash itself — and challenge #3.

Takeaway: after a framework jump, budget a runtime proof for every reflection-heavy library (mappers, IoC containers, serializers, mocking). "It builds" tells you nothing about them. And when a library is simultaneously broken, vulnerable and re-licensed — replace, don't chase versions.

2. One process, two repositories: the same-process rule

Applies to: combined & modules-only (it's the reason "modules-only" is still lockstep)

What made it hard

Assembly.LoadFile pulls every module into the host's process. Host and modules therefore share one resolved version of every common assembly — EF Core, Microsoft.Extensions.*, Newtonsoft, and the host's own DLLs. Two repositories, dozens (potentially hundreds) of csprojs, and any version disagreement compiles fine and fails only when the type actually crosses the boundary at runtime. The nastiest instance was latent in the HintPaths:

- ..\RecoApi\...\bin\Debug\net6.0\RECO.API.Domain.dll   ← framework BAKED IN
+ ..\RecoApi\...\bin\$(Configuration)\$(TargetFramework)\RECO.API.Domain.dll

After a retarget, the old paths would still resolve the stale net6 DLLs sitting on disk — modules build green against last year's host and detonate at load time. The failure (TypeLoad/MissingMethod, or subtler behavioural drift) points nowhere near "your csproj has net6.0 hard-coded in a path".

How it was resolved

Three mechanisms, all enforced by automation rather than discipline: (1) HintPaths rewritten to $(Configuration)\$(TargetFramework) tokens so host and module cannot disagree again; (2) package versions aligned everywhere — and in the modules-only tooling the target versions are derived from the host's own csproj files at run time, making the host the single source of truth instead of a hard-coded table; (3) a lockstep guard: the modules migration refuses to run at all if any host project still targets the old TFM, because modules built against an unmigrated host are worse than no migration.

Evidence

Module loading was verified by a harness doing Assembly.LoadFile + full type resolution of every module against the net10 host binaries (44–66 types per module, providers found) — the strongest cheap proof that the process-sharing contract held.

Takeaway: in any plugin architecture, treat "which versions may exist in the process" as a single system-wide invariant with a single source of truth — never as N independent csproj decisions. And grep your HintPaths for baked-in TFMs before a retarget, not after the first TypeLoadException.

3. The identity clone: when the generated code is correct and your assumption isn't

Applies to: combined & modules-only (any module with a same-type map) · Class: silent data corruption

What made it hard

The repository layer snapshots "old content" before updating a tracked entity, so it can compare old vs new. Under AutoMapper, a same-type map produced a copy. Mapperly, asked for the equivalent TblSynonym Clone(TblSynonym source), generates — perfectly reasonably, from its point of view:

internal static partial Models.TblSynonym? Clone(Models.TblSynonym? source)
{
    return source == null ? default : source;   // ← the SAME reference, not a copy
}

An aliased snapshot makes every old/new comparison silently equal. No exception, no log line — change detection just stops working. This is the most dangerous class of migration bug: both libraries are behaving correctly; only the semantic contract between them differs.

How it was resolved

Never declare same-type Mapperly maps for cloning. Clone = new instance + the generated in-place copy:

.Register<Models.TblCode, Models.TblCode>(source =>
    { var target = new Models.TblCode(); RecoMappings.CopyOnto(source, target); return target; })

Caught, again, by the empirical parity harness — specifically by comparing object references, not just values, for the clone shape.

Takeaway: when swapping a library, enumerate the semantic contracts call sites rely on (copy vs reference, ordering, null handling, culture) and test those explicitly. Reading the generated code of a source generator is not paranoia; it's the whole point of using one.

4. Drift before the first change: establish a green baseline or debug fog forever

Applies to: combined (and to any modules repo that lagged its host)

What made it hard

Before any framework was touched, two of the four modules did not compile against the current host: the host's RelType model had lost the IDataRecordWithRelationships marker interface that the modules' generic pipelines require (CS0311). Had the retarget started on top of that, every subsequent error would carry the question "upgrade breakage, or was it always broken?" — multiplied across every module.

How it was resolved

Repair drift first (a one-line interface declaration, matching every sibling model), record a green net6 baseline, and only then retarget. In the two-repo world this generalises to a rule: the modules repo must compile against the host repo's current main before either starts migrating.

Takeaway: a migration inherits every latent inconsistency and makes it look like its own fault. Spend the first day proving the pre-state is green — it's the cheapest debugging you'll ever buy.

5. It's never the language — it's the ecosystem

Applies to: combined (host-side); modules feel it only as package alignment

What made it hard

.NET 6 → 10 spans four annual releases, yet not one line of application logic needed rewriting for the runtime itself. Everything that broke was the ecosystem moving several majors at once:

BreakShapeCost

How it was resolved

A package matrix decided once (host and modules identical — challenge #2), the sink rewritten behind an unchanged config contract, and everything else mechanical. The discipline that mattered: pin the mapper during Part A (no mapping churn while retargeting) and take the ecosystem hit in one reviewed sweep.

Takeaway: inventory your custom extensions of third-party libraries (sinks, providers, formatters) first — that's where multi-major jumps land hardest. And treat "worked on net6, dies on first DB call" encryption defaults as a standing item on every SqlClient-era upgrade checklist.

6. Environment coupling: absolute paths meet an API that demands them

Applies to: combined (host config); a portability win for every future dev machine

What made it hard

appsettings pointed the module folders at an absolute path from the original developer's machine (C:\Work\...\Modules\Filters). The naive fix — make them relative — crashes, because Assembly.LoadFile rejects relative paths outright. So the system was stuck: unportable config, and an API that punishes the obvious cure.

How it was resolved

Relative paths in config, resolved once, at a defined anchor, inside the loader:

// ModuleProviderLoader ctor
_pathToModulesFolder = Path.GetFullPath(pathToModulesFolder, AppContext.BaseDirectory);
Takeaway: "make it relative" is only half a fix — the other half is choosing the anchor (AppContext.BaseDirectory, not the process CWD) and resolving at one choke point instead of at every consumer.

7. Doing it four hundred times: scale, batching and the lockstep cutover

Applies to: modules-only, primarily — the reason it exists as a separate track

What made it hard

The reference repo has four modules; production reality is hundreds of subscribers owned by multiple people on different schedules. Hand-editing csprojs at that scale guarantees drift (one missed HintPath, one stale package pin), and the same-process rule (#2) turns any single miss into a service-down event at the shared restart. Meanwhile the AutoMapper conversion (#1) is per-module human work that cannot be big-banged.

How it was resolved

Split the work by automatability, not by module:

  • Part A (mechanical, scripted): discovery-based tooling that sweeps every csproj — TFM, HintPath tokens, versions derived from the host — idempotently, with dry-run default, per-file backups, rollback, a batching filter (-ModuleFilter 'RDBES.*'), and a build gate that collects failures instead of stopping, so one red module can't hide the state of the other 399.
  • Part B (intent-bearing, human): the script only inventories it — a CSV of every AutoMapper reference, which becomes the tracking sheet. One reference conversion first, then fan out; identical subscriber shapes make repetition mechanical.
  • Cutover: branches can be prepared over weeks, but host switch + all module deployments land in one restart window. The lockstep guard makes the wrong order impossible to script by accident.
Takeaway: at scale, the deliverable is not "migrated modules" — it's a repeatable migration function plus a worklist. Draw the automation boundary where intent begins: scripts transcribe facts; humans transcribe intent; and anything the script won't do it should still detect and report.

8. The meta-challenge: migration tooling you can actually trust

Applies to: both tracks — anyone writing the automation · Platform: Windows PowerShell 5.1

What made it hard

The scripts themselves had to be resilient, and the platform fought back with failure modes that only appear at runtime, in specific environments:

  • Under $ErrorActionPreference='Stop', PS 5.1 turns any redirected native stderr line into a terminating error — including git checkout announcing success ("Switched to a new branch") on stderr. The script worked in a console and crashed in CI.
  • Get-ChildItem -LiteralPath … -Recurse -Include '*.cs' silently ignores the -Include filter — the inventory scanned every file (including its own previous output CSV, self-amplifying the count 357 → 775) until replaced with explicit extension checks.
  • StrictMode + PowerShell's array unrolling: a one-element result becomes a scalar with no .Count; a single-line file from Get-Content ditto.
  • PS 5.1 parses BOM-less UTF-8 as ANSI — em-dashes in string literals decode to bytes that include a curly closing quote, terminating strings mid-sentence with surreal parse errors. ASCII + BOM, always.

How it was resolved — and, more importantly, how it was verified

Each pitfall got a structural fix (an EAP-relaxing wrapper for all native calls; extension filters; call-site @(); ASCII+BOM). But the real answer was the verification method: reproduce the known-good result from scratch. The scripts were run against disposable clones of the pre-migration state and their output diffed against the hand-migrated reference branch — byte-identical csprojs, then idempotency re-run (0 changes), rollback restore, the lockstep guard, and a full RED→fix→GREEN build-gate rehearsal that reproduced the exact failure cascade a real user would see (one EmailSink compile error → every module failing with CS0246 'RECO') and its recovery.

Takeaway: a migration script is a program that edits programs — test it like one. If a hand-verified reference migration exists, "re-derive it byte-for-byte from a clean checkout" is the strongest cheap test available, and rehearsing the failure path is what makes the runbook's troubleshooting section true rather than hopeful.

What deliberately did not change

Half the craft was restraint. The architecture survived untouched — no rewrite to minimal APIs, no AssemblyLoadContext adventure, Newtonsoft kept, LINQ-era patterns kept: Assembly.LoadFile plugin loading, Generic Host + Startup bootstrapping, the controller surface, the persistent queue. Every deliberate change beyond the retarget lived on its own branch (net10-upgrade first, then replace-automapper-mapperly) so that any regression stays attributable to exactly one diff. The one honestly-open item was stated as such: full end-to-end runtime verification requires the dev SQL Server, which the migration machine could not reach — so it shipped as an explicit checklist for a DB-connected machine, not as a claim.

The one-slide summary

If you remember five things
1. Runtime-prove your reflection-heavy libraries after a framework jump — green builds are silent about them.
2. In a plugin system, versions are one process-wide invariant — tokenise HintPaths, derive module versions from the host, guard the lockstep.
3. Baseline before you migrate; separate drift-fix, retarget, and library-replacement into attributable diffs.
4. Draw the automation boundary at intent: script the mechanical sweep (idempotent, dry-run, rollback), inventory the human work, never auto-write what encodes business meaning.
5. Verify the tooling by reproducing the known-good result — and rehearse the failure path so your troubleshooting table describes reality.
